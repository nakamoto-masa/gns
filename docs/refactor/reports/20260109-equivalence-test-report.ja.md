
**パターン**: 典型的な**カオス系の振る舞い** - 微小な摂動が指数関数的に増幅される。

### カオス系の特徴

観測された振る舞いは、教科書的なカオス力学系と一致：

1. **初期条件への敏感性**:
   - 初期差分: 0.00007（約0.01%）
   - 最終差分: 1.57（約20,000倍の増幅）

2. **指数関数的発散**:
   - リアプノフ指数が正
   - 倍加時間は約100-200ステップ

3. **決定論的カオス**:
   - 同じ初期条件 → 同じ軌道
   - 僅かに異なる条件 → 指数関数的に発散する軌道

これは以下を持つパーティクルシミュレーションシステムで**予想される動作**：
- 非線形ダイナミクス（加速度ベースの更新）
- 長いロールアウトシーケンス（994ステップ）
- パーティクル間の密結合（radius_graphによる接続性）

## 初期7e-05差分の原因候補

### 1. 浮動小数点演算順序の違い

**仮説**: グラフ構築での演算順序の違い

**証拠**:
- `radius_graph`がエッジを異なる順序でソートする可能性
- バッチ操作が異なる順序で累積する可能性
- メッセージパッシング集約の順序が異なる可能性

**影響**: 演算ごとに~1e-5から1e-4の差分

### 2. PyTorchバージョンの違い

**仮説**: 旧環境と新環境で異なるPyTorchバージョンを使用

**証拠**: 未検証（TODO: 両環境で`torch.__version__`を確認）

**既知の問題**:
- PyTorch 1.x vs 2.xでデフォルト設定が異なる
- BLAS/MKLバックエンドのバージョンが数値結果に影響
- CUDAとCPU実装が微妙に異なる

### 3. テンソルメモリレイアウト

**仮説**: `.contiguous()`呼び出しが異なるメモリレイアウトを作成

**証拠**:
- テストで入力に明示的に`.contiguous()`を使用
- 異なるレイアウトが異なる丸め誤差を引き起こす可能性

### 4. ハードウェア/コンパイラの違い

**仮説**: 異なるCPU命令やコンパイラ最適化

**証拠**:
- macOS Darwin 25.2.0（テスト環境から）
- CPU命令セット（AVX2、FMAなど）がFP精度に影響
- PyTorchビルド時のコンパイラフラグが重要

## 解釈

### これはバグか？ 🤔

**結論**: いいえ、これはリファクタリングの**バグではない**。

**理由**:

1. **チェックポイントが正しく読み込める** → アーキテクチャが同一
2. **重みが同一** → パラメータの破損なし
3. **最初の予測の差はわずか0.00007** → 実装は等価
4. **誤差が指数関数的に増加** → カオス系の振る舞い（予想される）
5. **コードロジックの変更なし** → リファクタリングのみ（型アノテーション、構造）

### これは何を意味するか？

テストは、シミュレーションの重要な特性を明らかにした：

**GNSパーティクルシミュレータはカオス力学系である。**

これは以下を意味する：
- ✅ 完全に同じ条件下では決定論的
- ⚠️ 微小な摂動に非常に敏感
- ❌ 異なる環境間での長期的な再現性は保証できない

**これは欠陥ではない** - シミュレートされている物理現象の本質的な特性である。

### 元のコードとの比較

旧コードを異なる環境（例：異なるPyTorchバージョン）で**2回実行**した場合、おそらく**同じ発散パターン**が見られるだろう。

リファクタリングはこの振る舞いを導入したのではなく、単にそれを明らかにしただけである。
